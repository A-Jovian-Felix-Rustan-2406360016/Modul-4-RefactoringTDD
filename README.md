Reflection Module 2 :
1. Selama proses mengerjakan modul 2 ini, khususnya di bagian exercise, saya menemukan beberapa isu code quality yang menjadi hambatan utama saya dalam menyelesaikannya. Masalah pertama yang ditemukan yaitu mengenai test coverage. Awalnya, test coverage saya masih terbilang cukup rendah karena memang belum saya implementasikan semua di modul sebelumnya. Namun, setelah saya sudah menambahkan unit test dan merasa sudah semuanya telah di cover, hasil tes jacoco stuck di 98% untuk class ProductRepository. Hal itu membutuhkan waktu yang lumayan lama untuk saya menemukan jalur logika yang belum saya tes. Meskipun line coveragenya terlihat hijau, tetapi sebenarnya terdapat percabangan tersembunyi yaitu method create yang menangani ID produk, method findById, dan method edit. Strategi saya untuk masalah ini hanyalah mencoba menggunakan konsep positive dan negative testing untuk memperluas kemungkinan skenario yang mungkin terjadi, dan akhirnya saya berhasil mendapatkan coverage 100%.

Selain itu, saya juga mendapat beberapa masalah mengenai code quality setelah saya melakukan code scanning menggunakam bantuan sonarcloud. Kebanyakan isu tersebut sebenarnya hanya masalah kecil tetapi tetap ditandai oleh sonarcloud untuk best practice. Beberapa issue yang sudah saya selesaikan diantaranya :
- Empty Method Issue : Terdapat method yang isinya kosong atau belum diimplementasi. Strategi yang saya gunakan untuk ini yaitu dengan menambahkan komentar yang berisi informasi tentang method tersebut. Walaupun method tersebut empty, tetapi masih diperlukan saat kita jalankan aplikasi kita sehingga tidak bisa dihapus begitu saja.
- Grouping Dependencies Issue : Penulisan dependencies yang cenderung acak, tidak dikelompokkan sesuai karakternya. Walaupun hal ini terlihat tidak penting, tetapi penulisan dependencies yang berkelompok sesuai fungsinya dapat mempermudah orang membaca code kita dan salah satu prinsip clean code. Strateginya yaitu saya tinggal refactor atau mengelompokkan ulang setiap dependency.
- Unused Import Issue : Terdapat beberapa library yang sebenarnya tidak digunakan di code saya, tetapi saya tetap import. Hal ini memang tidak mempengaruhi dari sisi fungsionalitas, tetapi akan membuat code menjadi berantakan dan melanggar prinsip clean code. Strateginya yaitu saya mencari library apa saja yang memang tidak digunakan dan menghapusnya.
- Throwing Exception Issue : Pada beberapa bagian code, saya menambahkan throw exception untuk menghindari adanya potensi kesalahan logika. Namun, dalam implementasi methodnya, saya tidak benar-benar melempar exception tersebut sehingga membuat code menjadi berantakan. Hal ini sama seperti unused import library karena akan melanggar prinsip clean code. Strateginya yaitu saya tinggal hapus throw exception pada method yang memang tidak benar-benar mengimplementasikannya.

2. Implementasi pipeline CI/CD yang telah dibuat dalam module ini menurut saya telah memenuhi definisi dari Continuous Integration dan Continuous Deployment. Dari sisi CI, alur kerja yang telah memastikan bahwa setiap push atau pull request di github, sistem otomatis menjalankan testing. Ini penting karena memastikan perubahan baru tersebut tidak merusak fungsionalitas yang sudah ada. Jadi, integritas kode tetap terjaga. Lalu, integrasi dengan jacoco juga memberikan feedback berupa coveragenya. Sementara itu, untuk Continuous Deployment, proses deploy sudah sepenuhnya otomatis. Ketika kode sudah lolos tahap CI, maka sistem langsung melakukan build dan deploy ke lingkungan server. Secara keseluruhan, alurnya menjadi lebih cepat, konsisten, dan dapat diandalkan.



Reflection Module 3 : 
1. Dalam module ini, saya telah menerapkan beberapa prinsip SOLID melalui proses refactor yang telah dilakukan. Pertama, saya menerapkan Single Responsibility Principle (SRP) dengan menghapus inheritance antara CarController dan ProductController. Sebelumnya, hubungan inheritance tersebut membuat CarController ikut terikat pada tanggung jawab yang seharusnya tidak relevan. Dengan memisahkannya, maka setiap class controller kini berdiri sendiri dengan setiap controller hanya punya satu tanggung jawab yaitu terhadap entitasnya. Kedua, saya membuat interface untuk CarRepository dan ProductRepository sebagai bentuk penerapan Open Closed Principle. Dengan adanya interface ini, maka sistem akan terbuka untuk ekstensi seperti produk jenis baru atau mobil jenis baru, tetapi tetap tertutup terhadap modifikasi kode yang sudah ada. Ketiga, saya tidak mengimplementasikan secara langsung Liskov Substitution Principle karena memang tidak ada pewarisan yang membutuhkan substitusi di module ini, sehingga kode ini tidak melanggar dan sudah memenuhi LSP. Keempat, kode ini memang sudah implementasi Interface Segregation Principle dari sananya sehingga tidak perlu modifikasi. Hal ini terlihat dari semua method interface yang memang dimanfaatkan oleh setiap class yang implement. Terakhir, saya menggunakan interface pada ProductController dan CarController sebagai bentuk penerapan Dependency Inversion Principle. Dengan ini, maka komponen tingkat tinggi tidak lagi bergantung pada concrete class, melainkan pada abstraksi atau dalam kasus ini interface. Hal yang sama juga diterapkan pada layer service dimana implementasi service menggunakan interface ProductRepository dan CarRepository. Ini membuat dependency lebih fleksibel.

2. Penerapan prinsip SOLID membuat struktur kode menjadi lebih modular dan fleksibel sehingga gampang untuk dimaintain jangka panjang. Salah satu manfaatnya, dengan pemisahan tanggung jawab yang jelas antar kelas, setiap komponen memiliki peran spesifik dan tidak saling tumpang tindih. Dampaknya, ketika terjadi perubahan kebutuhan atau penambahan fitur, kita tidak perlu mengubah banyak bagian sekaligus. Risiko kesalahan atau error juga lebih kecil karena area perubahan menjadi terisolasi dan terkontrol. Desain seperti ini membuat proses pengembangan terasa lebih stabil, terutama ketika proyek mulai bertambah kompleks. Contohnya, setelah CarController tidak lagi inherit dari ProductController, maka semua perubahan pada fitur produk tidak akan mempengaruhi fitur mobil dan sebaliknya. Lalu, dengan penggunaan interface pada ProductRepository dan CarRepository, saya bisa mengganti mekanisme penyimpanan data, misalnya dari in memory menjadi database, tanpa mengubah layer lain. Pendekatan ini membuat sistem menjadi lebih fleksibel terhadap perubahan implementasi.

3. Sebaliknya, tanpa penerapan SOLID, kode akan cenderung memiliki struktur yang sulit dikembangkan. Ketika dependency tidak dikelola dengan baik dan bergantung secara langsung pada concrete class, perubahan kecil akan menimbulkan efek yang domino sehingga bisa mempengaruhi banyak bagian sistem. Sebagai contoh, controller yang langsung bergantung pada implementasi konkret repository tanpa menggunakan interface, saat ada perubahan maka saya harus ubah banyak kode sekaligus yang meningkatkan potensi kesalahan dan memperlambat proses pengembangan. Dalam jangka panjang, desain tanpa prinsip SOLID akan membuat sistem semakin kompleks dan sulit dipahami. Testing juga menjadi lebih sulit karena komponen saling bergantung erat dan tidak dapat diuji secara terisolasi. Akibatnya, biaya maintenance meningkat karena setiap perubahan membutuhkan effort lebih besar dibandingkan jika sistem sejak awal dirancang dengan struktur yang modular dan abstraksi.
